# Contributing Guide

Thank you for contributing to QA Interview Bible.

This repository is a public, employer-facing knowledge base. Every contribution must be safe to share publicly and valuable to readers outside any single company context.

## Contribution principles

- Write for long-term reuse, not for a single company or project.
- Prefer durable engineering concepts over tool hype or private workflow detail.
- Generalize all examples, incidents, screenshots, and data.
- Keep the tone professional, precise, and educational.
- Optimize for clarity, structure, and teaching value.

## Confidentiality rules

Do not include:

- employer names in sensitive contexts
- internal architecture details
- internal process documents
- customer data
- production logs
- access credentials, tokens, keys, or secrets
- regulated, contractual, or proprietary material

If a real experience inspired an article, rewrite it into a generalized teaching example.

## Scope of acceptable content

Good contributions include:

- testing concepts and heuristics
- automation patterns
- debugging frameworks
- quality strategy
- leadership lessons that are broadly applicable
- interview preparation material
- generalized case studies

Avoid contributions that are mainly:

- company-specific implementation notes
- private process documentation
- vendor marketing summaries
- unverified AI-generated filler

## Required article structure

Every article should include these sections unless there is a strong reason not to:

1. Summary
2. Main content
3. Interview questions
4. Generalized example
5. References

## File naming

- Use lowercase letters only.
- Separate words with hyphens.
- Avoid spaces and special characters.
- Prefer concise, searchable names.

Examples:

- `api-testing-basics.md`
- `mobile-release-checklist.md`
- `bug-reproduction-guide.md`

## Suggested authoring workflow

1. Pick one topic per page.
2. Start with the teaching goal and target reader.
3. Draft from first principles.
4. Add generalized examples.
5. Add interview-oriented takeaways where relevant.
6. Review against the style, security, and AI rules.

## Review checklist

Before submitting, confirm that the page:

- uses Markdown only
- covers one clear topic
- uses a lowercase filename
- follows the style guide
- contains no confidential or proprietary information
- uses generalized examples
- includes practical and educational value
- is suitable for a public engineering audience
